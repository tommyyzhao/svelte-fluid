## Next session focus: Continuous random splat generation

### Goal
Add a parameter for continuous random splat generation as an effect. For example, the "drops in water" (InkInWater) preset should have random splats appearing continuously, simulating water drops.

### Key files to read

1. **src/lib/engine/FluidEngine.ts**
   - `randomSplats(count)` method (around line 696)
   - `tick()` / `update()` loop (around line 685)
   - How splats are created and queued

2. **src/lib/Fluid.svelte**
   - `buildConfig()` function
   - `$effect()` that propagates prop changes
   - How config flows from props → engine

3. **src/lib/presets/InkInWater.svelte**
   - Current configuration (curl=5, densityDissipation=0.4, etc.)
   - How presetSplats are defined

4. **src/lib/engine/types.ts**
   - `FluidConfig` interface
   - Where to add new config fields

5. **docs/decisions/0005-hot-update-buckets.md**
   - Hot-update bucket system (A/B/C/D)
   - Where the new parameter should go (likely Bucket A or B)

### Design considerations

1. **New config fields** (tentative):
   - `randomSplatRate: number` — splats per second (0 = disabled, default)
   - `randomSplatCount: number` — splats per burst (default 1)
   - `randomSplatInterval: number` — ms between bursts (alternative to rate)

2. **Implementation approach:**
   - Option A: Add a timer in `tick()` that triggers `randomSplats(1)` at intervals
   - Option B: Add a `$effect` in Fluid.svelte that uses setInterval/clearInterval
   - Option A is cleaner (single source of truth in engine)

3. **Hot-update bucket:**
   - Rate/count changes should be hot-updatable (Bucket A)
   - This means `setConfig()` needs to handle timer changes

4. **Preset integration:**
   - InkInWater: `randomSplatRate={0.5}` (one drop every 2 seconds)
   - Other presets: leave at 0 (no continuous splats by default)

5. **Performance:**
   - Use `performance.now()` for timing, not frame counting
   - Respect `paused` flag (don't queue splats when paused)
   - Consider `lazy` prop (don't splat when not visible)

### Questions to answer

1. Should rate be "splats per second" or "interval in ms"? (leaning toward rate for UX clarity)
2. Should the splat count per burst be configurable, or always 1?
3. Should presets use this, or keep their hand-crafted presetSplats + let users add rate?
4. How does this interact with `initialSplatCount`? (initial = one-time at mount, rate = continuous)
